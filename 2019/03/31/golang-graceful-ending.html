写在前面

按照一般的设计原则， 每个 HTTP 请求都是独立存在的，因此一般情况下 Web 应用都很容易做水平扩展；同时这也意味着 HTTP 请求发起重试的成本是很低的，从而使得大家在开发 Web 接口的时候很少关注应用的优雅中止。

不过，业务中 ① 总会存在对中止比较敏感的接口（比如支付相关），并且 ② 总会存在一些带状态的服务，此时优雅中止就显得比较重要了。

本文通过一个Go 定时任务示例来简单介绍优雅中止的处理思路。

适用人群

入门——初级√——中级——高级；本文适应初级及以上。

浅谈 Go 应用的优雅中止

优雅中止的含义

所谓“优雅中止”，是指应用接收到特定的中止信号（比如 INT、TERM）后，不再接受外部的新请求，也不再创建内部的新任务，保持应用进程一直到旧需求和旧任务执行完成后再终止退出。

Kubernetes 中 Pod 的终止机制

k8s 定义了终止 Pod （业务进程在 Pod 中运行）的基本步骤：当主动删除 pod 时，系统会在强制终止 Pod 之前将 TERM 信号发送到每个容器中的主进程，过一段时间后（默认为 30 秒），再把 KILL 信号发送到这些进程。除此之外， k8s 通过钩子方法提供了对 容器生命周期 的管理能力，允许容器启动后和终止前执行一些特殊的命令。

当应用被打包进镜像且运行在 k8s 中的时候，如果应用实现了优雅中止的机制，就可以充分地利用 k8s 提供的高可用功能，在应用升级（发新版本）和 Pod 管理（宿主机维护时把 Pod 漂移到另一个宿主机，或者流量闲时动态收缩 Pod 的数量从而把资源省出来另作他用）的过程中实现服务的零中断。

优雅中止的 Go 代码示例

package main

import (
	"fmt"
	"os"
	"os/signal"
	"syscall"
	"time"
)

func main() {
	c := make(chan os.Signal)
	// Go 不允许监听 SIGKILL/SIGSTOP 信号
	// 参考 https://github.com/golang/go/issues/9463
	signal.Notify(c, syscall.SIGINT, syscall.SIGTERM)

	second := time.NewTicker(time.Second)
	minute := time.NewTicker(time.Minute)

A: // 由于 for-select 嵌套使用，设置跳出 for 循环的标记
	for {
		select {
		case s := <-c:
			// 收到 SIGTERM/SIGINT 信号，跳出 for 循环结束进程
			fmt.Printf("get signal %s, graceful ending...\n", s)
			break A
		case <-second.C:
			go mySecondJobs()
		case <-minute.C:
			go myMinuteJobs()
		}
	}
	fmt.Println("graceful ending")
	// 做一些操作让异步任务正常结束，这里偷懒地采取简单等待的方式 😆
	time.Sleep(time.Second * 10)
	fmt.Println("graceful ended.")
}

func mySecondJobs() {
	tS := time.Now().String()
	fmt.Printf("starting second job: %s \n", tS)
	time.Sleep(time.Second * 1) // 假设每个任务消耗 1 秒时间
	fmt.Printf("second job %s are done. \n", tS)
}
func myMinuteJobs() {
	tS := time.Now().String()
	fmt.Printf("starting minute job: %s \n", tS)
	time.Sleep(time.Second * 2) // 假设每个任务消耗 2 秒时间
	fmt.Printf("minute job %s are done. \n", tS)
}

源码解读-处理思路

  1. 通过 signal.Notify 捕获特定的信号；
  2. 通过 for + select 来实现循环任务，同时检测 signal.Notify 欲捕获的信号；
  3. 当发现指定的信号时，跳出 for 循环，并采取一定措施结束异步任务。

源码解读-值得关注的几个点

  1. 代码中采用了 go mySecondJobs() 和 go myMinuteJobs() 异步任务的方式，如果采用同步的方式将无法捕获信号（主线程在处理业务逻辑，没有空闲处理信号捕获逻辑）；
  2. 源码中偷懒地采取简单等待的方式来保证异步任务正常结束，非普适方法，实际开发中需要根据情况做定制；
  3. time.Ticker 的使用是有注意事项的（参考文献中最后一篇有讲到），不过上面的代码不会触发这个问题。

小结

默认情况下，Go 应用在接收到 TERM 信号后直接退出，如果此时有过程没处理完（接收到请求后尚未返回响应，异步任务尚未结束），则会导致过程的异常中断，影响服务质量。通过在代码中显式地捕获 TERM 信号及其他信号，能够感知到操作系统对进程的处理，从而主动采取措施优雅地结束应用进程。

随着 k8s 的普及，考虑到期对进程生命周期的规范化管理，应用支持代码级的优雅中止（尤其是容器化的应用）有必要成为一种开发规范，值得引起每一位开发者的注意。
